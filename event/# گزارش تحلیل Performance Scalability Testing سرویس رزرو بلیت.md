

سناریوی تست شامل زنجیره کامل عملیات بوده است و صرفاً یک درخواست تکراری نبوده است:

```text
Get Event
   ↓
Reserve Ticket
   ↓
Get Reserve Status
   ↓
Payment
   ↓
Cancel Reservation
   ↓
Delete Reservation
   ↓
Transaction Controller (کل سناریو)
```

بار تست در چهار سطح مختلف اجرا شده است:

* 10 کاربر همزمان
* 25 کاربر همزمان
* 50 کاربر همزمان
* 100 کاربر همزمان

# 2- تحلیل زمان پاسخ

## میانگین پاسخ

میانگین زمان پاسخ کل سناریو در تمامی سطوح بار تقریباً ثابت و حدود 1ms باقی مانده است که نشان می‌دهد:

* سیستم در بارهای پایین و متوسط دچار اشباع نشده است.
* منابع CPU و Memory هنوز دارای ظرفیت آزاد هستند.
* افزایش تعداد کاربران اثر محسوسی روی Average Response Time نگذاشته است.

---

## تحلیل P95

P95 به معنی این است که 95 درصد درخواست‌ها سریع‌تر از این مقدار پاسخ داده شده‌اند.

| کاربران | P95  |
| ------- | ---- |
| 10      | 7 ms |
| 25      | 6 ms |
| 50      | 7 ms |
| 100     | 7 ms |

مشاهده می‌شود که:

* P95 تقریباً ثابت باقی مانده است.
* افزایش کاربران باعث افزایش محسوس تاخیر برای اکثر کاربران نشده است.
* سیستم تا 100 کاربر همزمان رفتار تقریباً خطی دارد.

---

## تحلیل P99

| کاربران | P99   |
| ------- | ----- |
| 10      | 12 ms |
| 25      | 11 ms |
| 50      | 16 ms |
| 100     | 13 ms |

در تست 50 کاربر جهش کوچکی در P99 مشاهده می‌شود:

```text
10 User  → 12ms
25 User  → 11ms
50 User  → 16ms
100 User → 13ms
```

این رفتار معمولاً ناشی از موارد زیر است:

* رقابت Threadها برای Connection Pool
* Garbage Collection
* افزایش Lock Contention روی دیتابیس
* صف شدن برخی درخواست‌ها در لایه Application

از آنجایی که در تست 100 کاربر این عدد دوباره کاهش پیدا کرده است، احتمالاً Bottleneck دائمی وجود ندارد و این افزایش ناشی از نوسان طبیعی سیستم بوده است.

# 3- تحلیل Throughput

| کاربران | Throughput  |
| ------- | ----------- |
| 10      | 61 req/s    |
| 25      | 95.3 req/s  |
| 50      | 186.3 req/s |
| 100     | 221.4 req/s |

روند رشد:

```text
10 User  -> 61 req/s
25 User  -> 95 req/s
50 User  -> 186 req/s
100 User -> 221 req/s
```

افزایش Throughput تقریباً خطی است ولی از 50 به 100 کاربر رشد کمتر می‌شود:

```text
95 → 186   (+95%)
186 → 221  (+18%)
```

این موضوع اولین نشانه نزدیک شدن سیستم به ظرفیت اشباع است.

# 4- تحلیل Bottleneck ها

## Bottleneck اصلی: Transaction Controller

در تمامی تست‌ها بیشترین زمان پاسخ مربوط به کل تراکنش بوده است.

| کاربران | Avg | P95 | P99 | Max |
| ------- | --- | --- | --- | --- |
| 10      | 8   | 12  | 16  | 16  |
| 25      | 7   | 13  | 14  | 14  |
| 50      | 9   | 16  | 40  | 40  |
| 100     | 8   | 14  | 17  | 56  |

مشخص است که:

* عملیات منفرد مانند Reserve یا Payment تقریباً زیر 2ms هستند.
* افزایش زمان پاسخ ناشی از مجموع عملیات و وابستگی میان درخواست‌هاست.
* زمان انتظار بین عملیات و ارتباط با دیتابیس سهم اصلی را دارد.

---

## Bottleneck احتمالی دیتابیس

نشانه‌ها:

* افزایش Max Response Time از 16ms به 56ms
* افزایش Std Dev در بارهای بالاتر
* رشد کمتر Throughput نسبت به رشد کاربران

احتمالاً علت یکی از موارد زیر است:

* Connection Pool کوچک
* Lock روی رکوردهای رزرو
* رقابت برای Update موجودی بلیت‌ها
* Transaction Isolation Level بالا

# 6- الگوهای خرابی یا افت عملکرد

هیچ الگوی خرابی مشاهده نشده است اما چند نشانه اولیه از نزدیک شدن به محدودیت ظرفیت وجود دارد:

### افزایش Maximum Response Time

```text
16ms → 14ms → 40ms → 56ms
```

### کاهش نرخ رشد Throughput

```text
95 req/s → 186 req/s → 221 req/s
```

این رفتار معمولاً نشان‌دهنده ورود تدریجی سیستم به ناحیه Saturation است.

### برای کل سناریو:

* Long Tail Latency مشاهده نشده است.
* بخش کوچکی از درخواست‌ها دچار تاخیر غیرعادی نشده‌اند.
